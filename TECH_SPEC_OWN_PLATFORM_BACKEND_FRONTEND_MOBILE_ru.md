# Техническое ТЗ на собственную реализацию
## Проект: Pilates Space — без GymMaster (Backend + Frontend + Mobile)

**Версия:** 1.0 (draft)  
**Дата:** 2026-03-02  
**Статус:** На согласовании

---

## 1) Цель

Разработать собственную платформу Pilates Space для управления:
- клиентами,
- расписанием и бронированиями,
- пакетами занятий и посещениями,
- оплатами (JCC Smart + наличные),
- уведомлениями (push),
- web + mobile self-service.

Исключить зависимость от GymMaster.

---

## 2) Область реализации

## 2.1 Backend
- API для web/mobile/admin.
- Бизнес-логика бронирования и оплаты.
- Управление продуктами (разовое, пакет 10/20/30, индивидуальные).
- Учет посещений и списание пакетов.
- Интеграция с JCC Smart.
- Обработка callback/webhook от JCC.
- Аудит действий и отчетность.

## 2.2 Frontend (Web)
- Публичный сайт студии.
- Личный кабинет клиента.
- Админ-интерфейс оператора.

## 2.3 Mobile (iOS/Android)
- Клиентский кабинет.
- Запись/отмена.
- Покупка и оплата.
- Push-уведомления.

---

## 3) Роли и права

1. **Суперадмин**
   - управление справочниками, тренерами, локациями, правами.
2. **Администратор студии**
   - клиенты, бронирования, продажи, отметки посещений, касса.
3. **Тренер**
   - просмотр своих групп, отметка факта посещения (опционально).
4. **Клиент**
   - просмотр расписания, бронь/отмена, покупки, история.

RBAC обязателен.

## 4) Функциональные модули

## 4.1 CRM клиентов
- Создание/редактирование профиля клиента.
- Контакты, согласия, примечания.
- Статус клиента (active/inactive/blocked).
- История посещений, покупок, платежей.

## 4.2 Расписание и группы
- Управление шаблонами классов (тип, длительность, емкость).
- Календарь занятий по локациям/тренерам.
- Лимит мест + waitlist.
- Правила отмены (cutoff-окно).

## 4.3 Бронирования
- Запись клиента на занятие.
- Проверка доступности мест.
- Автопроверка права записи (есть пакет/разовая покупка/долг).
- Отмена с применением правил возврата/штрафа.

## 4.3.1 Waitlist (обязательно в MVP)
- При заполнении группы клиент может встать в лист ожидания.
- Очередь waitlist ведется по времени добавления (FIFO), с возможностью приоритетов для админа.
- При освобождении места система автоматически предлагает место первому в очереди.
- Время на подтверждение места: 15 минут (Assumption, настраиваемо).
- Если клиент не подтвердил место в срок — предложение уходит следующему в очереди.
- Администратор может вручную перемещать/удалять клиентов из waitlist с записью в audit log.

## 4.4 Продукты и пакеты
- Продукты:
  - разовое групповое,
  - пакет 10/20/30,
  - индивидуальная тренировка.
- Срок действия пакета.
- Правила списания (по посещению/по типу класса).
- Ручные корректировки с аудитом.

## 4.5 Оплаты
- Методы: JCC Smart, наличные.
- Продажа на ресепшене и онлайн.
- Статусы транзакций и сверка.
- Чеки/квитанции (email/PDF — опционально в MVP).

## 4.6 Уведомления
- Push (обязательно в MVP):
  - booking_confirmed,
  - booking_cancelled,
  - waitlist_joined,
  - waitlist_spot_offered,
  - waitlist_spot_expiring,
  - waitlist_spot_confirmed,
  - waitlist_spot_missed,
  - class_reminder,
  - class_changed,
  - payment_success,
  - package_low_balance.

Требования к push-модулю:
- Поддержка iOS (APNs) и Android (FCM).
- Шаблоны уведомлений с локализацией EN/RU.
- Дедупликация уведомлений (не отправлять дубли при ретраях).
- Лог доставки (sent/delivered/failed, где доступно).
- Retry policy при временных ошибках провайдера.

## 4.7 Отчеты
- Выручка по дням/локациям/типам продуктов.
- Посещаемость, no-show, загрузка групп.
- Продажи пакетов и остатки.

## 5) Ключевые пользовательские сценарии

1. Клиент регистрируется, бронирует занятие, получает подтверждение.
2. Клиент покупает пакет онлайн, получает баланс посещений.
3. Админ продает пакет на ресепшене (кэш/JCC), пакет сразу активен.
4. После занятия выполняется списание одного посещения из пакета.
5. Клиент отменяет занятие в допустимый срок — место освобождается.
6. При отмене вне срока применяется правило штрафа (если включено).
7. Если группа заполнена, клиент встает в waitlist и получает push при появлении места.
8. Клиент подтверждает место из waitlist в отведенное время, иначе слот передается следующему.

## 6) Backend API (минимум)

- `POST /api/v1/auth/signup`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`

## 6.2 Catalog
- `GET /api/v1/locations`
- `GET /api/v1/trainers`
- `GET /api/v1/products`

## 6.3 Classes/Bookings
- `GET /api/v1/classes`
- `POST /api/v1/bookings`
- `POST /api/v1/bookings/{id}/cancel`
- `GET /api/v1/bookings/my`
- `POST /api/v1/waitlist/join` (classSessionId)
- `POST /api/v1/waitlist/{id}/confirm`
- `POST /api/v1/waitlist/{id}/leave`
- `GET /api/v1/waitlist/my`

## 6.4 Packages/Visits
- `GET /api/v1/packages/my`
- `GET /api/v1/visits/my`
- `POST /api/v1/admin/visits/mark`

## 6.5 Payments
- `POST /api/v1/payments/jcc/create`
- `POST /api/v1/payments/cash/confirm` (admin)
- `GET /api/v1/payments/{id}/status`
- `POST /api/v1/webhooks/jcc`

## 6.6 Admin
- `POST /api/v1/admin/clients`
- `POST /api/v1/admin/purchases`
- `GET /api/v1/admin/reports/revenue`
- `GET /api/v1/admin/reports/attendance`

## 8) Платежный процесс (JCC + кэш)

## 8.1 JCC
1. Создание purchase.
2. Создание платежа в JCC.
3. Оплата клиентом.
4. Callback/webhook от JCC.
5. Подтверждение транзакции.
6. Активация покупки/пакета.

## 8.2 Наличные
1. Создание purchase администратором.
2. Прием наличных.
3. Подтверждение `cash` в системе.
4. Активация покупки/пакета.

Требования:
- идемпотентность операций,
- ручной перевод в `manual_review` при расхождениях,
- ежедневная сверка транзакций.

## 14) Критерии приемки

1. Система полностью покрывает текущие ручные процессы студии.
2. Администратор выполняет продажу и запись без бумажных карточек.
3. Клиент самостоятельно записывается/отменяет через web/mobile.
4. Оплаты JCC и кэш корректно отражаются в балансе клиента.
5. Посещения корректно списываются из пакетов.
6. Отчеты по выручке и посещаемости доступны в админке.

---

## 15) Что не входит в MVP

- интеграция с Revolut;
- сложная loyalty/referral система;
- продвинутая маркетинг-автоматизация;
- бухгалтерская ERP-интеграция.

---

## 16) Рекомендованный стек

- Backend: NestJS + PostgreSQL + Redis
- Web: Next.js
- Mobile: React Native
- Queue jobs: BullMQ
- Observability: Sentry + Grafana/Prometheus
- CI/CD: GitHub Actions
