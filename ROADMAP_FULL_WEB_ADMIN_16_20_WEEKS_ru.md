# ROADMAP (FULL) — Web + Admin, 1 разработчик
## Pilates Space (собственная платформа, без mobile на этапе 1)

**Горизонт:** 16–20 недель  
**Режим:** 1 full-time разработчик  
**Scope:** backend + web кабинет + кастомная админка + JCC POS init + waitlist + push + прод-запуск

---

## Общие принципы
- Работа спринтами по 1 неделе.
- В конце каждой недели: демо + фиксация риска + обновление плана.
- Каждая фича считается готовой только при: код + тесты + логирование + роли + базовая документация.

---

## Week 1–2: Discovery и фундамент

### Цели
- Финализировать требования и контракты.
- Подготовить архитектуру и проектную базу.

### Задачи
- Утвердить BRD/ТЗ (текущий scope).
- Подготовить ERD и доменную модель.
- Подготовить OpenAPI v0 (черновые endpoint'ы).
- Поднять окружения: dev/staging.
- Базовый каркас монорепо (api/web/admin).
- Настроить CI/CD (lint/test/build/deploy).

### Артефакты
- `ERD_v1`
- `OpenAPI_v0`
- `Architecture overview`

### Milestone
- Проект собирается и деплоится в staging.

---

## Week 3–5: Backend Core v1

### Цели
- Реализовать основную доменную логику без платежей.

### Задачи
- Auth (JWT + refresh), RBAC.
- Клиенты (CRUD, статусы, история).
- Локации, тренеры, шаблоны занятий.
- Сессии занятий и емкость групп.
- Бронирования/отмены с cutoff-правилами.
- Продукты и пакеты (10/20/30/разовые/инд).
- Ledger списаний/начислений посещений.
- Audit log каркас.

### Milestone
- Админ может создать клиента, расписание, продажу (без онлайн-оплаты), записать клиента на занятие.

---

## Week 6–7: Web кабинет + публичный сайт (v1)

### Цели
- Дать клиенту self-service на web.

### Задачи
- Публичные страницы: Home/About/Trainers/Gallery/Contacts.
- Login/Signup/Reset.
- Расписание, бронь/отмена.
- Профиль клиента.
- Баланс пакетов + история посещений.
- Базовая локализация EN/RU.

### Milestone
- Клиент может зарегистрироваться и сам записываться/отменять на сайте.

---

## Week 8–10: Кастомная админка (операционный контур)

### Цели
- Закрыть ежедневные задачи администратора студии.

### Задачи
- Dashboard (сегодняшние группы, загрузка, оплаты).
- Клиенты: поиск, карточка, заметки, история.
- Расписание: создание/редактирование сессий, лимиты мест.
- Продажи: разовое/пакеты/индивидуальные.
- Касса: операции кэш + статусы платежей.
- Ручные корректировки пакетов (с обязательным reason).
- Аудит-экран операций.

### Milestone
- Администратор может вести студию в админке без бумажных карточек.

---

## Week 11–12: JCC интеграция + POS initiation

### Цели
- Реализовать карточные платежи через JCC с запуском суммы на терминал из админки.

### Задачи
- JCC connector (create/status/callback).
- Endpoint `terminal/initiate` из админки.
- Привязка `purchaseId` ↔ `providerPaymentId`.
- Fallback polling при недоставке callback.
- Статусы: paid/failed/manual_review.
- Автоактивация пакета/занятия после успешной оплаты.
- Журнал `payment_events`.

### Milestone
- Из админки отправляется сумма на POS, после оплаты покупка закрывается автоматически.

---

## Week 13: Waitlist Engine

### Цели
- Полноценный waitlist для заполненных групп.

### Задачи
- Join/leave/confirm waitlist.
- FIFO очередь + ручной приоритет админом.
- Auto-offer при освобождении места.
- Таймер подтверждения (15 мин, configurable).
- Передача слота следующему.
- Логирование переходов состояний.

### Milestone
- Waitlist работает end-to-end с автоэскалацией по таймеру.

---

## Week 14: Push Notifications

### Цели
- Стабильный контур уведомлений.

### Задачи
- Notification service (APNs/FCM abstraction на backend).
- События: booking, waitlist, class changes, payment success.
- Retry + дедупликация.
- Лог доставки и техпанель статусов.

### Milestone
- Пользователи получают ключевые push-события, статусы видны в админке/логах.

---

## Week 15–16: QA, UAT, hardening

### Цели
- Подготовить стабильный production запуск.

### Задачи
- Integration + E2E по критическим путям.
- Нагрузочный smoke test (расписание/бронирования/оплаты).
- Security baseline (rate limit, CORS, secret rotation policy).
- Мониторинг/алерты (Sentry + health checks).
- Backup/restore check PostgreSQL.
- UAT с администратором студии.

### Milestone
- Go/No-Go решение по прод-релизу.

---

## Week 17–20 (буфер и расширенный hardening)

Используется при реальном full-режиме:
- устранение UAT-замечаний,
- UX polishing админки,
- улучшенные отчеты,
- повышение отказоустойчивости,
- релизные итерации без риска для операций студии.

---

## Критерии готовности релиза

1. Полный цикл записи/отмены/списания работает без ручного учета на бумаге.
2. Продажа в админке закрывается через кэш или JCC POS initiation.
3. Waitlist автоматически обрабатывает освобождающиеся места.
4. Push-уведомления доставляются по ключевым событиям.
5. Есть аудит действий, платежные журналы и базовые отчеты.
6. Staging и production имеют воспроизводимый деплой-процесс.

---

## Основные риски

- Ограничения JCC API/терминального сценария.
- Переоценка объема UI для админки при 1 разработчике.
- Расширение scope в середине разработки.

### Антирисковые меры
- Freeze scope после Week 2.
- Жесткий change-control (новые фичи только в backlog Phase 2).
- Еженедельные демо и переоценка плана.
